# 角色牌数据系统

## 范围和目标

系统只建立角色专属卡牌及其普通闪、专属灵光一闪、神圣闪光和角色专属强化数据。中立牌和怪物牌不收录、不解析效果；
画面中的非目标牌只能返回 `unknown` 或 `unsupported`，不能被猜成某张角色牌。

最终识别结果必须同时包含卡牌身份、变体、当前费用、运行时状态、结构化效果、位置以及分字段置信度。
卡牌目录描述事实，效果规则引擎计算合法操作，战斗策略再决定是否使用；三者不能互相代替。

## 当前里程碑

目前已经实现：

- `CardDefinition`：角色牌的稳定身份、名称、类型、基础费用、目标和效果。
- `CardVariant`：普通闪、专属灵光一闪、神圣闪光和角色强化对基础牌的差异层。
- `EffectSpec`：触发条件和机器可读效果动作。
- `CardObservation`：实时识别输出的标准画面事实，包含实例、身份、变体、区域和分字段置信度。
- `CardCatalog`：严格加载、关系校验和变体实体化。
- `RuntimeCardIndex`：启动时预编译卡名、效果文字和变体只读索引，并隔离候选资料与审核目录。
- `CardCollectorTask`：一次保存一帧图像、同帧 OCR 证据和待审核标签，绝不操作游戏。

WGC 可以在游戏窗口被遮挡时直接采集完整 1280×720 客户区。由于 STOVE 游戏进程以管理员权限运行，
自动导航任务也必须以管理员权限启动；普通权限仍可捕获，但 Windows 会拒绝鼠标注入。

当前采集器保存全帧，还没有猜测卡牌边界。必须先取得真实样本，才能可靠确定详情页和战斗手牌的裁切区域。

## 首名试点角色

首名角色已确定为：

```text
内部 ID：haide_mali
简体名称：海德玛丽
繁体名称：海德瑪麗
客户端语言：zh-TW

```

内部 ID 使用中文拼音生成，不依赖未来可能变化的英文译名。初始截图来自角色“能力值”页面，只用于确认
角色、语言和分辨率，不作为卡牌训练样本。正式样本从左侧“卡牌”页面开始采集。

## 采集一份样本

1. 用 `scripts/run-debug.ps1` 启动程序并选中正确的游戏窗口。
2. 在游戏中打开目标角色的一张卡牌，保持画面稳定。
3. 在一次性任务中选择“采集一份角色牌样本”。
4. 选择当前场景和 `zh-TW`；角色 ID 填 `haide_mali`，卡牌和变体未知时可以先留空。多个同时生效的
   强化层使用逗号分隔。
5. 点击运行。任务读取一帧、遮挡右下角潜在 UID、对这一帧执行一次 OCR，然后结束。
6. 在任务信息中查看生成的 `manifest.json` 绝对路径。

默认输出位置：

```text
datasets/cards/inbox/YYYY-MM-DD/<sample-id>/
├─ frame.png
└─ manifest.json
```

稳定 ID 使用小写 ASCII：

```text
角色：character_id
基础牌：character_id/card_01
变体：character_id/card_01/epiphany_a
```

普通闪（也称中立闪、通用闪）和专属灵光一闪属于同一基础强化层，同一张牌只能选择其一；神闪是
独立附加层，可以叠加在基础牌、普通闪或专属灵光一闪上。例如同一张牌同时具有专属灵光一闪与神闪，
采集配置中填写：

```text
character_id/card_01/epiphany_a,character_id/card_01/divine_flash_a
```

数据库按固定层级叠加，避免为每种组合复制一整张牌。普通闪使用 `kind=common_flash`，专属灵光一闪使用
`kind=epiphany`；两者互斥，神闪使用 `kind=divine_flash` 并在其后附加。

中文卡名只放在本地化名称字段中，不作为主键。

## 验证数据

检查正式卡牌目录：

```powershell
.\scripts\cards.ps1 validate-catalog
```

检查样本清单结构、PNG 是否存在以及图片哈希是否被修改：

```powershell
.\scripts\cards.ps1 validate-samples
```

实际样本图片默认被 `.gitignore` 排除。只有经过人工核对的角色牌事实 JSON 才进入 `data/cards/characters`。
字段模板位于 `data/cards/characters/character.template.json.example`。

游戏局外无法预览的普通闪和神闪资料放在 `datasets/cards/reference`。网页截图只用于确定候选词条、层级和
后续实机采集目标，不参与训练，也不能直接提升为正式目录数据。

## 离线识别普通闪、专属闪和神闪

当前第一版识别链路不依赖图像训练集，而是组合三类稳定证据：

1. 在分辨率无关的相对区域内用卡名 OCR 确定基础牌。
2. 在卡牌效果区用简繁归一、数字保持的模糊匹配查找普通闪、专属闪和神闪词条。
3. 相同文字同时存在于普通闪和神闪池时，检测文字左侧的金色神闪徽记；详情页若显示明确的
   “通用闪光”或“物主之闪光”分栏，则优先使用分栏。
4. 灵光一闪分支的效果文本可能与基础卡面完全一致（例如万人的英雄分支 a 只是新增 `[連結]` 特性标签）。
   此类分支必须在 `epiphany.pending.json` 中标注 `required_trait_tags_zh_tw`；识别时除文本匹配外，
   还要求对应特性以独立括号标签的形式出现在卡面上（带括号，或位于效果文本上方），否则一律判为基础层。

识别结果按两层组合输出：基础层为 `base`、`common_flash` 或 `epiphany` 三选一，之后可再叠加一个
`divine_flash`。效果词典和网页证据仍保持 `pending`，识别器会把证据状态原样带入结果，不能把自动识别
等同于人工批准。

用已经采集的样本离线验证：

```powershell
.\scripts\cards.ps1 recognize-flash datasets/cards/inbox/2026-07-17/<sample-id>/manifest.json --ignore-label
```

不加 `--ignore-label` 时，如果样本清单已经标有 `card_id`，识别器会把它作为卡牌身份提示，但仍自行识别
闪光效果。输出 JSON 包含基础牌、组合后的 `variant_ids`、命中的 OCR 文字、位置、文字/图层置信度和歧义原因。

在游戏中直接识别时，打开 ok-chaos 的一次性任务，选择“识别当前卡牌闪光（只读）”并运行。任务只读取
当前窗口的一帧，在任务信息中显示卡牌名称、卡牌 ID、普通闪、专属灵光一闪、神闪、组合变体和置信度。
这一入口不会生成 `frame.png`；需要保留证据时仍应明确运行“采集一份角色牌样本”。

### 运行时索引和 CardObservation

任务第一次运行时会把两类来源编译成一个只读 `RuntimeCardIndex`：

- `datasets/cards/reference` 与 `datasets/cards/review` 提供待审核卡名、效果文字和变体候选，只能用于识别。
- `data/cards/characters` 提供人工审核后的卡牌和变体定义，是策略层唯一可用的事实来源。

每次识别依次产生三层结果：

```text
FlashRecognition（识别器细节）
        ↓
CardObservation（本帧可复用的标准画面事实）
        ↓
RuntimeCardResolution（审核目录准入与实体化结果）
```

`CardObservation` 不保存整张截图，也不复制字体或 OCR 模型；它保存当前帧中这张卡的身份、组合变体、
画面区域、运行时状态以及各字段置信度。候选词库即使高置信度命中，也只会得到“识别成功、策略不可用”；
只有基础牌和全部活动变体都存在于 `data/cards`，目录校验和层级组合也通过时，准入结果才是 `ready`，
并返回可供规则与策略读取的 `MaterializedCard`。

信息面板同时显示“识别状态”和“策略可用”。这两个字段不能合并：前者回答画面里看到了什么，后者回答
这些事实是否已经人工审核到足以驱动自动操作。当前海德玛丽正式目录仍为空，因此真实样本能够正确生成
`CardObservation`，但会被标记为“卡牌尚未进入审核目录”。

## 第一批数据怎么采

只选一名角色建立纵向闭环，不要一开始铺开全部角色：

1. 采集该角色每张基础牌的详情页标准画面。
2. 采集每个可见顿悟分支及专属强化后的详情页。
3. 在战斗中采集这些牌的普通、悬停、选中、减费和不可使用状态。
4. 保留一部分完全不参与调参的画面作为 `validation` 集。
5. 根据首批画面实现布局定位和多证据匹配，再决定是否需要训练小模型。

海德玛丽试点可以在角色卡牌列表中直接运行两个一次性任务：

- `自动采集海德玛丽卡牌详情`：遍历八个卡位并保存基础详情。
- `自动采集海德玛丽灵光一闪`：逐张检查右下角按钮，只为有按钮的卡牌保存点击后的五分支总览。

灵光一闪总览样本先只标注基础 `card_id` 和 `scene=epiphany`。一张总览同时包含五个候选，必须在
人工审核阶段按画面从左到右分配 `epiphany_a` 到 `epiphany_e`，采集任务不能自动猜分支标签。

采集结束后执行以下命令，把四张五分支总览切成 20 个独立的待审核样本：

```powershell
.\scripts\cards.ps1 split-epiphany
```

切分使用经过实图核对的固定五列区域，生成的 ID 包含来源总览和分支字母；命令可重复执行，已存在的
分支只会被校验，不会重复创建。裁剪结果仍是 `pending`，切分不等于批准入库。

采样时不要把中立牌或怪物牌标成角色牌。它们即使偶然进入画面，也只用于验证系统能拒绝未知目标。

5. 战斗手牌定位、跨帧实例跟踪和动态状态识别，最后才接入出牌策略。
